A Graph IS an Adjacency List — Not Two Separate Sets

G = (V, E) looks like two sets, but it is ALWAYS represented as ONE combined adjacency list. V and E are not stored separately. Each index in the array = a vertex. Each linked list at that index = the neighbors of that vertex.

Could also be an adjacency matrix — same principle applies. One structure encodes both V and E.

Exam Hint: "If asked 'what is the data structure for a graph?' — the answer is an adjacency list, not 'a set of vertices and a set of edges.'"

Simple Graph Rules — What Is and Is NOT Allowed

RuleDetail
✗ Self-edges: edge from A to ANOT allowed in simple graphs. All adjacency lists in this course are simple — no self-edges ever.
✗ Multi-edges: two edges between the same pair in the same directionNOT allowed in simple graphs. Duplicate edges (same u→v twice) are invalid.
✗ Thinking opposing directed edges (u→v AND v→u) are multi-edgesALLOWED. Having both u→v and v→u in a directed graph is valid — they are two different edges in opposite directions. This does NOT violate simple graph rules.

Undirected vs Directed: How Edges Are Encoded

Undirected: edge {u,v} stored TWICE — in adj[u] AND adj[v]. Total entries = 2m.

Directed: edge u→v stored ONCE — in adj[u] only. Total entries = m.

Finding all INCOMING edges of v in a directed graph requires scanning all other vertices' lists — O(n+m). This is why "looking backwards" is expensive and why we build G^R for SCC.

Exam Hint: "MC trap: 'How many times does edge (A,B) appear in an undirected adjacency list?' — Answer: 2. In a directed adjacency list for A→B: 1 (only in A's list)."

What Is a Metagraph?

The metagraph G_SCC is just ANOTHER adjacency list. It follows all the same simple graph rules (no self-edges, no multi-edges) and is ALWAYS a DAG. Each "vertex" in the array represents one SCC.

To find which original vertices belong to SCC metanode k: look up all v where ccnum[v] == k. The metagraph MAY initially have multi-edges between metanodes — drop duplicates.

Exam Hint: "The metagraph is ALWAYS a DAG — no cycles allowed. This is guaranteed by the definition of SCCs (if two SCCs had a cycle between them, they'd merge into one)."

Session 2 — DFS Behavior, CC NUM Semantics & Graph Manipulation
03/01 — TA Tim West (+ TAs Jamie McPeek, Aaron Miller)

CC NUM: Semantics Differ by Graph Type

Undirected: ccnum[u]==ccnum[v] means u and v are mutually reachable (connected).

Directed: ccnum[u]==ccnum[v] means u and v are in the same STRONGLY connected component (mutual reachability). DO NOT use ccnum[] for general one-way reachability on directed graphs.

  • ccnum[] lookup is O(1) — index into the array directly.
  • On directed graphs, ccnum[] from plain DFS is unreliable for connectivity. The SCC algorithm's ccnum[] output is what correctly identifies strongly connected groups.
Exam Hint: "Trap question: 'Are u and v in the same connected component in this directed graph?' — ccnum[] only tells you they are in the same SCC, not that a one-directional path exists between them."

DFS Starting Vertex — When It Matters and When It Doesn't

The starting vertex for DFS is OPTIONAL. Only pass one in when the specific starting point changes your answer. If you only need ccnum[], skip it — the result is identical regardless of starting vertex.

  • Use cases that REQUIRE a starting vertex: finding a specific s→t path, computing pre/post timestamps from a known source, running Bellman-Ford-style exploration from s.
  • Use cases where starting vertex is irrelevant: finding all connected components, getting all ccnum[] values, general cycle detection across the whole graph.
Exam Hint: "If you pass a starting vertex AND only use ccnum[], that is fine but unnecessary. Lose no points — it just adds detail you didn't need."

DFS Exploration Order is NOT Deterministic

MistakeCorrection
✗ Assuming DFS will take a specific path or include a specific edgeDFS exploration order is arbitrary beyond the starting vertex. Never write a solution that relies on DFS choosing a particular neighbor order or traversal path.
✗ Trying to "steer" DFS to explore in a specific directionInstead of manipulating DFS internals, manipulate the INPUT GRAPH so that the edge or path you need is the only valid option — an "unavoidable consequence of the input graph." Tim: "Make it unavoidable, not directed."
✗ Assuming DFS textbook exercises reflect real DFS behaviorTextbook exercises enforce deterministic exploration (e.g., alphabetical order) only to make solutions consistent for grading. This is NOT guaranteed behavior of the DFS black box.
✗ Writing a solution that depends on which subtree DFS explores after hitting a cycleAfter a back edge is detected, which subtree DFS backtracks to next is arbitrary. Your correctness argument must hold for ALL possible exploration orders.

Advanced: Forcing DFS Order via Adjacency List

You CAN influence DFS exploration order by pre-ordering your adjacency list — but you must be fully explicit about this in your solution writeup. It is NOT implicit.

DFS processes neighbors in the order they appear in the adjacency list. If you deliberately sort or arrange the adjacency list before passing it to DFS, you can control exploration order. However, this is advanced, uncommon, and requires explicit justification in your solution.

Exam Hint: "If a problem seems to require a specific DFS order, look first for a graph transformation that makes the correct behavior unavoidable. Adjacency list ordering is a last resort and requires full explanation."

SCC Second DFS — Why Order Matters

The second DFS in Kosaraju's SCC algorithm must process vertices in DECREASING post-order from the first DFS. The vertex with the highest post[] value from the first run is the starting point of the second DFS on G^R.

  • The first DFS on G gives post[] for all vertices.
  • Sort vertices by decreasing post[] value. This sorted order becomes the exploration order for the second DFS on G^R.
  • Each new Explore() call in the second DFS discovers exactly one SCC.
  • You do NOT modify DFS — you only control what order you pass vertices as starting points.
Exam Hint: "When explaining SCC in an exam answer: 'Run DFS on G to compute post[] values. Then run DFS on G^R processing vertices in decreasing post[] order.' — The decreasing post-order IS the black box input you pass in."

Session 3 — SCC Meta Graph, Universal Vertices & Exam Strategy
03/04 — Prof. Brito (Gerandy)

Critical Identity: Reversing vs. Meta Graph Order

You can reverse the original graph BEFORE or AFTER building the SCC meta graph — you get the SAME result either way.

meta(G^R) = (meta(G))^R

  • If asked to find SCCs of G^R, you can just take G_SCC and reverse all its edges — O(n+m), no second SCC run needed.
  • This identity is also why Kosaraju's algorithm works: running DFS on G^R is equivalent to running DFS on the reversed meta-structure.
Exam Hint: "Great multiple-choice trap: 'Do you reverse the whole graph or half?' The answer: it does not matter. Reverse before or after SCC — same result."

The "Universal Vertex" Problem (Linear Time)

Problem: Find all vertices v in a directed graph such that v can reach EVERY other vertex AND every other vertex can reach v. Two-directional reachability requirement.

Brute Force (Too Slow — Do Not Use)

BFS from every vertex, check if all n vertices are visited. Repeat on reverse graph. Intersect results. Runtime: O(n · (n+m)) — quadratic. Prof explicitly said: "that sounds too much."

Optimal Solution — O(n+m)

  • 1. Run SCC to build G_SCC = (V_SCC, E_SCC).
  • 2. Run Topological Sort on G_SCC. Get topo order V1, V2, ..., Vk.
  • 3. Forward pass: Track which metanodes can reach ALL other metanodes looking forward in topo order. Vi can reach everything forward iff Vi is the UNIQUE SOURCE in the subgraph {Vi, Vi+1, ..., Vk}.
  • 4. Backward pass: Reverse G_SCC. Run the same procedure — find unique sources looking backward.
  • 5. Intersection: Vertices satisfying BOTH conditions are universal. Map back using ccnum[].

Runtime: O(n+m) — SCC is O(n+m), Topo Sort is O(n+m), both passes are O(n_SCC + m_SCC) = O(n+m).

Key DAG Theorem

Let G be a DAG with topological order V1, V2, ..., Vn. Then: Vi reaches Vj (for j > i) if and only if Vi is the unique source in the subgraph induced by vertices {Vi, Vi+1, ..., Vn}.

This converts an O(n+m) BFS reachability check into a single structural inspection of the DAG.

Exam Hint: "If an exam question gives you a DAG and asks about reachability, think about this theorem. 'Can vertex V reach all others?' ↔ 'Is V the unique source in the DAG?'"

Kruskal's — Must Be Connected

Kruskal's REQUIRES a guaranteed connected input graph. Cannot safely run on a disconnected graph in this class.

Some sources say Kruskal's can run on disconnected graphs and output a spanning forest. In THIS CLASS: keep it simple — Kruskal's input must be connected. You must guarantee connectivity from the problem statement or verify it (run BFS/DFS first) before calling Kruskal's.

  • Valid workaround: build connected subgraphs separately, run Kruskal's on each, then combine the resulting subtrees into a final solution.
  • Output format: edges[] is a list of edges [E1, E2, ..., E_{n-1}]. NOT parent pointers. Key difference from Prim's.
Exam Hint: "If a problem gives a potentially disconnected graph and asks for an MST, state how you guarantee connectivity before calling Kruskal's."

Kruskal's vs Prim's Output — Critical Difference

Kruskal's output = edges[] (edge list). Prim's output = prev[] (parent pointers). NOT interchangeable without conversion.

To traverse the MST after Kruskal's, you must first convert edges[] to an adjacency list in O(n+m). With Prim's, you can trace paths directly via prev[].

  • Prim's prev[] is identical in format to Dijkstra's prev[] and BFS prev[] — trace paths the same way.
Exam Hint: "Exam trap: using Kruskal's edges[] directly as a graph. Always state: 'Convert Kruskal's edges[] to adjacency list G_MST in O(n+m)' before running BFS/DFS."

Union-Find Internals (How Kruskal's Works)

  • Kruskal's is GREEDY: sorts all edges lightest to heaviest, adds each if it doesn't create a cycle. For each edge (v,w): if v and w are in different components → add. Same component → skip (cycle).
  • Union-Find is INTERNAL to Kruskal's — not a separate black box you call. Tim: "that is more of a data structure topic." You do not implement it for exams. The cycle check is abstracted as c(v) ≠ c(w).
  • Path compression: always attach a new vertex to the root with the highest existing tree depth. When merging, shallower root becomes child of deeper root. Keeps the union-find tree flat for efficiency.
  • Example (5.2B): edges chosen by weight order. {A,B} (w=1), {F,G} (w=1), extend to {F,G,D}, {F,G,D,H}. Skip any edge that creates a cycle. At weight 3: one edge creates cycle — skip. At weight 4: take one, skip the cycle-former. At weight 5: would form cycle — skip.
